## Provenance

*(Added in v0.4.0-alpha. Append this section to `docs/artifacts.md`, after the "Relationships" section and before "Lifecycle status".)*

Relationships (above) tell you the graph shape — which artifact led to which. Provenance tells you the full generation story for a single artifact, recorded on that artifact's own metadata:

```typescript
interface ArtifactProvenance {
  generatedBy: {
    agentId: string;
    capabilityIds?: string[];
    workflowId?: string;
  };
  inputArtifactIds: string[];
  frameworkVersion: string;
  createdAt: number;
}
```

```typescript
const artifact = await ctx.artifacts.create({
  // ...
  provenance: {
    generatedBy: {
      agentId: "product",
    },
    inputArtifactIds: [strategyArtifact.metadata.id],
    frameworkVersion: BUSINESS_OS_VERSION,
    createdAt: Date.now(),
  },
});
```

`provenance` is **optional** on `ArtifactMetadata` — artifacts created before v0.4.0-alpha have no provenance and `artifact.metadata.provenance` is simply `undefined` for them. Nothing about the frozen `ArtifactManager` interface changed; this is a purely additive field.

### Why provenance and relationships both exist

They answer different questions and are populated independently:

- **Relationships** (`linkRelationship` / `getRelationships`) are the queryable graph — "what did this artifact produce, and what produced it" as an indexed, traversable structure spanning the whole artifact store.
- **Provenance** is the artifact's own self-description — "what exactly generated *me*, from what inputs, using which agent and capabilities, at what framework version." It's denormalized onto the artifact itself so a single `get()` call answers lineage questions without a graph traversal.

In practice these should always agree for the immediate parent/child hop (an artifact's `provenance.inputArtifactIds` should match the source side of its `generated` relationship), but they're recorded through two separate calls (`create()`'s `provenance` field and a separate `linkRelationship()` call) — an agent author needs to populate both, they aren't derived from each other automatically.

### On `update()`

`UpdateArtifactInput` also accepts an optional `provenance` field. If omitted, the new version **carries forward** the previous version's provenance rather than clearing it — an update that doesn't specify new provenance is assumed to be a straightforward edit of the same lineage, not a re-generation from scratch. Pass an explicit `provenance` to `update()` only when the new version was genuinely regenerated from different inputs.

### Inspecting provenance from the CLI

```bash
bos artifacts
```
```
[prd] PRD: ai meeting notes
  id: prd-fcabe88c  version: 1  status: draft
  project: acme  updated: ...
  generated by: agent:product
  derived from: strategy-2bcfbc98
```

### `frameworkVersion`

Sourced from `BUSINESS_OS_VERSION` in `@business-os/artifact-core` (`packages/artifact-core/src/version.ts`). Bump this constant alongside git tags so provenance records stay accurate about which framework version produced them — this is a manual step, not automated from `package.json` or git.
