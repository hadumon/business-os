# CLI (`bos`)

`bos` is the terminal entry point for BusinessOS, shipped from `packages/cli`. It's the primary way to actually run agents — `defineAgent` and `Runtime` are the framework's programmatic API, but `bos` is how a person uses BusinessOS day to day.

## Installing

From the repo root, after building:
```bash
pnpm --filter @business-os/cli build
pnpm --filter @business-os/cli exec pnpm link --global
```
Then `bos` is available anywhere. Verify with `bos --version`.

Without linking, invoke it directly: `node path/to/business-os/packages/cli/dist/index.js <command>`.

## Commands

### `bos init`
Creates a `.business/` workspace in the current directory.
```bash
bos init --project acme
```
Creates:
```
.business/
  artifacts/
  memory/
  state/
  business.config.json
```
`.business/state/artifacts.sqlite` is **not** created here — it's created lazily on the first artifact write. This is normal; `bos doctor` reports it informationally rather than as a failure.

### `bos doctor`
Checks environment and workspace health: Node version, presence of `.business/`, config readability, subdirectory existence. Exits non-zero only on genuine problems.

### `bos run <agent>`
Runs a registered agent.
```bash
bos run discover --project acme --topic "AI-powered restaurant inventory management"
bos run strategy --project acme --topic "..." --from <discover-artifact-id>
bos run product  --project acme --from <strategy-artifact-id>
```

Flags:
- `-p, --project <name>` — required for all agents.
- `-t, --topic <topic>` — required for `discover` and `strategy`.
- `-f, --from <artifactId>` — links/consumes an upstream artifact. For `strategy`, this is optional (a Discover artifact to link via `linkRelationship`). For `product`, this is **required** (the Strategy artifact to derive a PRD from).
- `--from latest` — resolves automatically to the most recently updated artifact in the project, instead of requiring you to copy an id by hand.

The CLI validates that `--from` refers to an existing artifact before running anything, so a typo'd or stale id fails fast rather than after burning a full agent run.

### `bos artifacts`
Lists artifacts in the workspace, optionally filtered:
```bash
bos artifacts --project acme
bos artifacts --type prd
```
Each entry shows id, version, status, and — if present — provenance (`generated by`, `capabilities`, `derived from`). See [`artifacts.md`](./artifacts.md#provenance).

### `bos memory`
Lists memory documents (not artifacts) in the workspace, optionally filtered by `--path` prefix or `--keyword`.

### `bos workflow <agent>`
Prints the effective execution pipeline for an agent, for human inspection — not a live introspection of a running `Workflow` instance, just a static description of stages in order. Useful for understanding what an agent does before running it.

## Known agent registry

As of v0.4.0-alpha, `bos run` recognizes: `discover`, `strategy`, `product`. Registering a new agent with the CLI means adding it to `AVAILABLE_AGENTS` in `packages/cli/src/commands/run.ts` and to the stage list in `commands/workflow.ts` — this is manual today; there's no dynamic agent-discovery mechanism yet.

## Design notes

- **`bos` never talks to the network.** All state lives in `.business/` on disk. This means workspaces are portable and inspectable with ordinary tools (`cat`, a text editor, `sqlite3` if you want to query metadata directly).
- **Every command is independently runnable.** There's no persistent CLI session or server process — each invocation opens its own `Memory`/`ArtifactManager`/`Runtime`, does its work, and exits.
- **The `node:sqlite` experimental warning is suppressed** at the CLI entry point (`packages/cli/src/index.ts`) so it doesn't clutter output on every command. This doesn't affect the API — Node still emits the warning if you use `node:sqlite` directly.
